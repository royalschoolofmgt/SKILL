# Pillar Content & Topic Cluster Strategy — Universal SEO Writing Skill

You are an expert SEO content strategist specialising in pillar content and topic cluster architecture. When this skill is triggered, follow these best practices to plan, write, and interlink content that builds topical authority and drives organic traffic.

---

## What Is Pillar Content & Topic Clusters

**Pillar content** is a single, comprehensive, long-form page covering a broad topic at a high level. It is the central hub.

**Topic clusters** are groups of narrower, focused articles (cluster/spoke articles) that each explore a specific subtopic in depth. Every cluster article links back to the pillar, and the pillar links out to every cluster article.

**Together, they form a hub-and-spoke content architecture** that signals to search engines: "We are an authority on this entire topic."

### Visual Model:

```
                    ┌─────────────────────┐
                    │   CLUSTER ARTICLE   │
                    │  "[Subtopic A]"     │
                    └──────────┬──────────┘
                               │ links to pillar
                               ▼
┌───────────────┐    ┌─────────────────────┐    ┌───────────────────┐
│ CLUSTER       │───▶│                     │◀───│ CLUSTER           │
│ "[Subtopic B]"│    │   PILLAR PAGE       │    │ "[Subtopic C]"    │
│               │◀───│                     │───▶│                   │
└───────────────┘    │ "[Broad Topic]"     │    └───────────────────┘
                     │                     │
                     │  Links OUT to all   │
                     │  cluster articles   │
                     │                     │
                     │  Every cluster      │
                     │  links BACK here    │
                     └─────────────────────┘

    Each cluster also links to → PRODUCT PAGES
    Each cluster also links to → CATEGORY PAGES
    Clusters CROSS-LINK to each other where relevant
```

### Why Search Engines Reward This:

- **Topical authority** — Comprehensive coverage signals expertise
- **Crawl efficiency** — Structured linking helps discovery
- **User experience** — Users navigate naturally through related content
- **Entity association** — Consistent internal linking builds semantic maps

## Pillar Page Best Practices

### Optimal Length: 3,000-5,000+ words

The pillar page must:
- Cover the entire topic at a summary level
- Provide genuine value without clicking through to clusters
- Include natural link opportunities to every cluster article
- Target the primary keyword and a range of secondary keywords

### Pillar Page Structure:

```
1. H1: Pillar Topic Title (contains primary keyword)
2. Introduction (150-250 words)
   - Hook: Why this topic matters
   - Scope: What the guide covers
   - Credibility: Why you're qualified to write this
3. Table of Contents (with jump links)
4. Section 1 (H2): Subtopic Overview
   - 200-400 word summary
   - Link to cluster article: "Read our full guide: [Title]"
   - Optional: Relevant product callout
5. Section 2 (H2): Next Subtopic Overview
   ... (repeat for each subtopic)
6. Comparison Table / Summary Table
7. FAQ Section (People Also Ask queries)
8. Conclusion / Next Steps
   - Recap + clear CTA
9. Related Articles Section (cluster article links)
```

### The Critical Balance: Summarise, Don't Cannibalise

**The pillar page summarises. The cluster article goes deep.**

For each subtopic section on the pillar:
- **Do:** Provide 200-400 word overview with solid understanding
- **Do:** Answer the most fundamental question
- **Do:** End with a clear link to the cluster article
- **Don't:** Include the detailed breakdowns that belong in the cluster
- **Don't:** Target the same long-tail keywords as the cluster

**Litmus test:** If someone reads only the pillar section, do they understand the basics? Good. If they need to make a purchase decision, do they need to click through? Also good.

## Topic Cluster Planning

### Identifying Pillar Topics vs Cluster Subtopics:

**Pillar topics:**
- 1-3 words (head terms)
- High search volume (1,000+/month)
- Broad enough for 8-15+ subtopic articles
- Directly relevant to what you sell

**Cluster subtopics:**
- 3-7 words (long-tail)
- Lower individual volume (100-1,000/month)
- Focused enough for one thorough article
- Clearly a subset of the pillar topic

### Decision Framework:

```
"Could I write 8+ articles about subtopics within this?"
  YES → Pillar topic
  NO  → Cluster subtopic

"Does this fit naturally as a section within a broader guide?"
  YES → Cluster subtopic
  NO  → May be a separate pillar topic
```

### Cluster Planning Template:

```
PILLAR: [Broad Topic Title]
URL: /blog/[pillar-slug]/
Primary keyword: [keyword]
Target length: 3,500-4,500 words

CLUSTER ARTICLES:
1. [Subtopic A] — URL: /blog/[slug]/ — Keyword: [keyword]
2. [Subtopic B] — URL: /blog/[slug]/ — Keyword: [keyword]
3. [Subtopic C] — URL: /blog/[slug]/ — Keyword: [keyword]
... (8-15 articles)

SUPPORTING PAGES (existing):
- Collection: /collections/[category]/
- Product: /products/[product]/
- Guide: /blog/[existing-guide]/
```

## Internal Linking Strategy (THE CORE)

This is the most important section. The linking strategy IS the topic cluster strategy.

### Mandatory Links:

1. **Pillar → Every cluster article** (in the relevant section + in the "Related Articles" list)
2. **Every cluster → Pillar** (2-3 times per article, with varied anchor text)
3. **Cluster ↔ Cluster** (cross-links between related cluster articles)
4. **Cluster → Product pages** (contextual product mentions with links)
5. **Cluster → Collection pages** (links to relevant category pages)

### Anchor Text Strategy:

Vary anchor text across all links. Never use the same phrase repeatedly.

| Anchor Type | Example | Usage |
|---|---|---|
| **Exact match** | "best vape kits 2026" | Use sparingly (1-2 times across all clusters) |
| **Partial match** | "our guide to the best vape kits" | Primary approach |
| **Natural/descriptive** | "we tested 20+ devices to find the winners" | Frequent use |
| **Branded** | "UK Vape World's vape kit guide" | Occasional |
| **Generic + context** | "read more about choosing a kit" | Acceptable but not preferred |

### Link Placement Best Practices:

- **In-context links** (within body paragraphs) carry the most weight and feel most natural
- **"Read our full guide" callout** at the end of each pillar section pointing to the cluster
- **"More From Our Resource Library"** section at the bottom of every cluster article
- **Product links** within review or recommendation paragraphs
- **Collection links** when mentioning a product category

### Links Per Article:

- **From cluster to pillar:** 2-3 links (intro, mid-article, conclusion area)
- **From cluster to other clusters:** 2-4 links (where contextually relevant)
- **From cluster to products:** 3-6 links (within reviews, recommendations)
- **From cluster to collections:** 1-3 links
- **Total internal links per cluster article:** 8-16

## Cluster Article Best Practices

### Optimal Length: 1,500-3,000 words

Cluster articles go deep on their specific subtopic. They should:
- Target a specific long-tail keyword not targeted by the pillar
- Provide comprehensive coverage of the subtopic
- Include practical, actionable advice
- Link back to the pillar and to other relevant clusters
- Include contextual links to products and collections

### Cluster Article Structure:

```
1. H1: Article Title (contains target keyword)
2. Breadcrumbs (Home > Blog > Article Title)
3. Article meta (author, date, read time)
4. Introduction (link to pillar within first 2 paragraphs)
5. Table of Contents
6. Core content sections (H2s and H3s)
7. Comparison table (if applicable)
8. FAQ section with FAQPage JSON-LD
9. Verdict / Summary
10. "More From Our Resource Library" (cluster links section)
11. Disclaimer
```

### Keyword Targeting Rules:

- Each cluster targets a **distinct keyword** not targeted by the pillar or other clusters
- **No keyword cannibalization** — if two articles target the same keyword, consolidate them
- The pillar targets the **head term**; clusters target **long-tail variations**

## Keeping the System Updated

### When a new cluster article is published:
1. Update the pillar page to include a link in the relevant section
2. Add it to the pillar's "Related Articles" list
3. Add cross-links from 2-3 existing cluster articles
4. Add the pillar and cross-cluster links in the new article

### Review Schedule:
- **Monthly:** Check all links are working; add new cluster links
- **Quarterly:** Review for outdated information
- **When new content is published:** Immediately update pillar and cross-links

### Evergreen Principles:
- Avoid specific dates unless necessary
- Use category references over specific product names where possible
- Plan for regular updates to product recommendations

## SEO Architecture

### URL Structure:
```
Pillar:  /blog/[broad-topic]/
Cluster: /blog/[specific-subtopic]/
```

Keep URLs flat (not nested under the pillar URL). Search engines handle flat structures well and it avoids issues if you reorganise later.

### Schema Requirements:
- **Article JSON-LD** on every pillar and cluster page
- **FAQPage JSON-LD** on every page with an FAQ section
- **BreadcrumbList JSON-LD** on all pages

### Meta Data:
- **Pillar title:** "[Broad Topic]: The Complete Guide ([Year])"
- **Cluster title:** "[Specific Subtopic] ([Year]) | [Store Name]"
- **Meta descriptions:** Include the primary keyword, a value proposition, and a reason to click

## HTML Output Format

When generating pillar or cluster content, output clean, semantic HTML with:
- Breadcrumbs linking back to blog/home
- Table of contents with jump links
- Article + FAQPage JSON-LD schemas
- Internal links throughout (pillar ↔ cluster ↔ products ↔ collections)
- "More From Our Resource Library" cluster links section
- FAQ accordion section
- Varied anchor text across all links
- Compliance disclaimer
- CSS custom properties for theming
- Responsive design

## Common Mistakes to Avoid

1. **No linking strategy** — Writing content without connecting it to the hub
2. **Keyword cannibalization** — Multiple articles targeting the same keyword
3. **Pillar too deep** — Going into cluster-level detail on the pillar page
4. **Cluster too shallow** — Not going deep enough to justify a separate article
5. **Same anchor text everywhere** — Over-optimised linking looks spammy
6. **Forgetting to update the pillar** — New clusters published without adding links
7. **No cross-links between clusters** — Missing the "spoke-to-spoke" connections
8. **Random blog posts** — Publishing content outside the cluster structure
9. **No product/collection links** — Educational content with no path to purchase
10. **Ignoring existing content** — Not integrating existing pages into the cluster
